# 硅基员工模板与模型设计

**Date:** 2026-04-08  
**Scope:** `SiliconPerson` 模板复制语义、模型快照、审批模式覆盖、模板库演进  
**Status:** Design baseline

## 背景

`硅基员工` 不是一次性配置出来的静态对象，而是可以从模板复制、再按当前配置初始化的独立实体。这里最重要的设计点有两个：

1. 模板复制必须是“人格复制”，不是“运行态复制”。
2. 模型绑定必须在保存后冻结成该员工自己的快照，不再跟随主模型自动联动。

这意味着 `SiliconPerson` 的模板系统要服务于“快速创建”和“长期稳定”两个目标：创建时尽量方便，创建后尽量独立。

## 目标

1. 定义模板复制的语义，确保复制后得到的是新的员工，而不是共享引用的壳。
2. 定义模型绑定快照的保存规则，避免员工配置被全局模型配置隐式影响。
3. 明确 `approvalMode` 的覆盖逻辑，支持 `inherit / always_ask / auto_approve` 三种模式。
4. 为未来模板库和模板版本演进预留入口，但不把模板系统做成过度复杂的平台。

## 非目标

1. 本设计不讨论复杂的模板市场或在线分享协议。
2. 本设计不把模板和运行历史混成一个对象。
3. 本设计不要求模板跟随主模型配置实时联动。

## 模板复制语义

模板复制采用“人格模板复制”。

### 复制输入

复制的输入通常来自两类来源：

- 一个已有的 `SiliconPerson`
- 一个模板来源配置，可能是系统模板、用户模板或当前配置生成的临时模板

### 复制输出

复制后的新员工必须满足：

- 复制 `person.json`
- 复制 `assets/`
- 重置员工标识
- 重置创建 / 更新时间
- 初始化空 `runtime.db`
- 不复制历史会话
- 不复制历史消息
- 不复制历史任务
- 不复制未读状态
- 不复制审批上下文
- 不复制任何运行态缓存

复制语义的关键不是“拷贝多少文件”，而是“哪些东西要继续保留成模板人格，哪些东西必须被清空成新实例”。

## 模型快照设计

`SiliconPerson` 需要保存自己的 `modelBindingSnapshot`。这个快照不是一个动态指针，而是一次明确的绑定结果。

### 设计原则

1. 创建时可以从当前配置继承。
2. 保存后必须冻结，成为该员工自己的快照。
3. 后续主模型变化，不应自动回写已创建员工。
4. 如果要更新模型绑定，必须通过显式编辑动作重新生成快照。

### 推荐快照内容

```json
{
  "source": "current_config",
  "provider": "openai-compatible",
  "model": "gpt-5.1",
  "reasoning": {
    "enabled": true,
    "effort": "medium"
  },
  "tooling": {
    "planningEnabled": true,
    "delegationEnabled": false
  },
  "frozenAt": "2026-04-08T00:00:00.000Z"
}
```

这里的重点是“快照”而不是“引用”。如果员工一直跟着主模型配置自动漂移，那么复制、回滚和审计都会变得不可控。

### 支持两种创建方式

1. **集成当前配置创建**
   - 创建时读取当前主模型配置
   - 立即写入 `modelBindingSnapshot`
   - 保存后变成该员工自己的静态绑定

2. **手动配置创建**
   - 由用户直接选择 provider、model、effort 或相关能力开关
   - 保存后同样冻结为独立快照

两种方式在保存后的结果必须一致：都变成 `SiliconPerson` 自己的绑定，不再引用外部可变配置。

## 审批模式覆盖

`approvalMode` 建议支持三种值：

- `inherit`
- `always_ask`
- `auto_approve`

### 语义建议

- `inherit`：沿用全局或上层默认策略。
- `always_ask`：每次涉及审批的动作都必须显式确认。
- `auto_approve`：在允许范围内自动通过审批，不再反复询问。

### 设计要点

审批模式是员工级策略，不应散落在多个界面或运行分支里。

建议把审批模式理解为一层覆盖关系：

1. 系统默认策略提供基础值
2. 当前配置可以决定创建时的初始值
3. `SiliconPerson` 保存自己的最终值
4. 后续运行按该员工自己的策略执行

这样做的好处是，模板复制时不会把审批逻辑弄成“共享状态”，同时也便于审计每个员工为什么会自动审批或总是询问。

## 与模板库的关系

当前方案不需要一开始就做成完整模板市场，但应该从一开始就把结构留成可演进形态。

### 第一阶段

- 以 `SiliconPerson` 自身作为模板单元
- 允许从当前配置创建新员工
- 允许基于已有员工复制新员工

### 第二阶段

- 增加系统模板目录
- 增加用户模板收藏
- 增加模板版本号和模板来源信息

### 第三阶段

- 形成可复用的模板库
- 支持模板预览、差异查看和模板导入导出
- 支持模板版本升级与兼容提示

模板库演进时要保持一个原则：模板是“创建入口”，不是“运行态主控”。运行态主控仍然属于 `SiliconPerson` 实例本身。

## 推荐字段边界

`person.json` 中与模板和模型相关的字段建议保持集中，避免散落成多个互相覆盖的配置点。

### 建议保留

- `approvalMode`
- `modelBindingSnapshot`
- `skills`
- `workflows`
- `featureFlags`
- `schemaVersion`
- `version`

### 建议冻结

- provider
- model
- reasoning 开关
- effort
- 计划能力开关
- delegation 能力开关
- 任何从当前配置“继承”而来的值

冻结的意思不是“不允许改”，而是“保存时写入自己的快照，之后不再自动跟随上游变化”。

## 复制与更新原则

### 复制

复制时应当以“模板人格复用”为目标：

- 可以保留员工人格描述
- 可以保留角色设定
- 可以保留技能和工作流定义
- 可以保留文件型资源
- 不保留历史运行数据
- 不保留当前进程态

### 更新

更新员工时建议把操作分成两类：

1. **人格更新**
   - 修改 `rolePersona`
   - 修改 `skills`
   - 修改 `workflows`
   - 修改 `approvalMode`

2. **模型更新**
   - 重新选择 provider / model
   - 重新生成 `modelBindingSnapshot`
   - 显式写入新的快照版本

这样可以避免“改人格”与“换模型”混成一个按钮，导致后续审计和复制语义不清。

## 未来模板库演进建议

未来如果要把模板系统做大，建议沿着下面方向演进：

- 模板来源可标记：system / user / copied
- 模板支持版本号和来源追踪
- 模板支持导入导出，但导出物仍然以 `person.json` 为核心
- 模板支持差异预览，方便用户理解复制后会得到什么
- 模板支持兼容提示，但不要把兼容逻辑塞进运行态

最重要的是，不管模板库怎么演进，`SiliconPerson` 仍然必须是最终运行对象，而不是模板引用本身。

## 风险与约束

1. 如果模型绑定不冻结，员工会随着全局配置漂移，复制语义会失真。
2. 如果审批模式到处覆盖，未来很难解释一次审批为什么自动通过。
3. 如果模板库过早做复杂，反而会破坏当前“从当前配置创建”的简单路径。

## 需要主线程确认的点

1. `modelBindingSnapshot` 是否需要额外记录“创建时来源配置 ID”。
2. `approvalMode` 是否允许未来扩展更多枚举值，还是先固定这三个。
3. 模板库第一版是否只做本地模板，不考虑跨设备同步。
4. `featureFlags` 是否需要区分产品级开关和员工级开关。
