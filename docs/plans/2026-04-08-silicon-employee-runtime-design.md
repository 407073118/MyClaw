# Silicon Employee Runtime Design

**Date:** 2026-04-08  
**Scope:** `desktop/employees` 的消息投递、会话运行时、审批流、状态灯、模型快照与数据流  
**Status:** Design baseline

## Background

这份文档描述硅基员工的运行时语义。这里的重点不是 UI 长什么样，而是消息如何进入员工私域、`currentSession` 如何维护、审批如何卡点、状态如何回写，以及主控台和私域之间的数据流如何保持单向清晰。

再次强调：`SiliconPerson` 不是 OpenClaw-style 的临时 subagent，而是长期存在的数字员工。运行时必须围绕“持续身份 + 持续会话 + 持续状态”来设计。

## Runtime Principles

### 1. 主控台负责投递，不负责消费全部结果

主聊天发送给员工的消息，只负责触发员工私域运行。主控台不自动承接完整执行正文，避免控制平面被结果洪水污染。

### 2. `currentSession` 是唯一默认入口

每个员工都维护自己的 `currentSession`。所有 `@` 消息都投递到这个会话，运行时不做复杂关联会话搜索，也不在多个 session 之间自动选路。

### 3. 状态灯是主控台与私域之间的契约面

主控台只接收员工状态投影：idle / running / needs_approval / done / error，再加上未读和待审批提示。完整上下文留在员工私域。

### 4. 配置快照一旦保存就冻结

员工可以选择“集成当前配置”来绑定模型或其它运行参数，但一旦保存成快照，就不再随着主模型的后续变化自动联动。之后如果要变化，只能手动修改员工自己的配置。

## Message Routing

### 1. 主聊天消息进入投递层

主聊天中的消息先做目标解析：

- 如果消息包含 `@某个员工`，则识别为员工投递消息
- 如果没有命中员工，则仍然是主控台对话，不进入员工私域

### 2. 投递到 `currentSession`

命中员工后，系统直接把消息写入该员工的 `currentSession`。如果该员工没有 `currentSession`，则自动创建默认 session，再把消息写入。

### 3. 不做跨 session 搜索

运行时不查找“最相关的旧会话”或“最近一次相似任务会话”。我们始终把当前消息投递到当前会话，因为这才符合长期员工系统的可预期性。

### 4. 会话空间是员工自己的

点击头像进入的，是员工的聊天空间，不是主控台的某个视图副本。用户看到的是员工自己的 session 历史、运行结果与状态演化。

## Approval Flow

每个 `SiliconPerson` 都可以单独设置审批模式：

- `inherit`
- `always_ask`
- `auto_approve`

运行时处理原则如下：

- `inherit`：继承当前上层配置或默认治理策略
- `always_ask`：遇到需要审批的动作时停住，等待用户确认
- `auto_approve`：在允许范围内自动继续，不额外打断

审批态进入后，右侧状态灯变为 `needs_approval`，并以黄色提示用户。审批通过后，运行继续；审批被拒绝则进入终止或错误分支，具体结果回写到该员工私域。

## Status Updates

状态灯更新不是 UI 装饰，而是运行时事件的投影。建议状态流如下：

1. 消息进入时标记 `running`
2. 遇到审批卡点标记 `needs_approval`
3. 成功结束标记 `done`
4. 失败或异常标记 `error`
5. 空闲回稳后标记 `idle`

同时，运行时需要维护未读与待审批提示，让主控台知道哪些员工有新结果、哪些员工正在等人拍板。

## Model Snapshot Semantics

员工模型绑定支持两种语义：

- 直接绑定一个明确模型配置
- “集成当前配置”生成一份快照

快照的关键规则是：它只在保存瞬间采样当前配置，之后不再跟随主模型自动变化。这样可以避免员工运行时因为上游配置漂移而失去稳定性。

运行时读取员工模型时，优先使用员工自己的快照或显式绑定，而不是回头看主模型当前状态。

## Tasks, Schedules, Workflows

`tasklist`、`schedules`、`workflows` 都应进入同一个员工运行时，而不是各自散落成不同系统。

- `tasklist`：作为员工待办或执行输入进入运行时
- `schedules`：作为定时触发源，按时间把任务注入员工私域
- `workflows`：作为结构化执行路径，在员工私域中展开并驱动工具或步骤

它们的共同原则是：都围绕员工的 `currentSession` 和私域状态运行，不直接绕过员工实体去打主控台。

## Control Plane vs Private Domain Data Flow

主控台与私域之间的数据流应保持清晰分层：

### 主控台到私域

- 用户在主聊天发起 `@` 指派
- 主控台把消息投递到目标员工的 `currentSession` 连接
- 如果需要，主控台只携带最少的路由信息和审批上下文

### 私域内部

- 员工会话记录完整消息、工具调用、审批状态和结果
- 员工的模型、skills、workflows、tasklist、schedules 在私域内形成可持续运行上下文

### 私域到主控台

- 只回传状态摘要、未读提示、待审批提示和完成信号
- 不自动回传完整结果正文
- 用户若要细看，进入员工私域或再次 `@` 追问

这套流向保证主控台轻量、私域完整，也让“指挥台”和“员工房间”两个概念在系统层面真正成立。

## Non-Goals

1. 不做复杂的跨会话检索与自动归并。
2. 不把完整结果自动灌回主控台消息流。
3. 不把硅基员工实现成一次性 subagent。
4. 不兼容旧的 `desktop/employees` 伪代码数据形态。

## Risks and Open Points

1. `inherit` 的上层来源需要主线程最终拍板，是继承主模型、主控台策略还是某个 workspace 默认值。
2. `done` 状态与“完成但还有未读结果”之间的边界需要统一，避免头像栏信号不清。
3. `workflows` 和 `tasklist` 的优先级关系需要主线程补一条明确规则，避免同一员工同时被多种输入源驱动时产生歧义。
4. `currentSession` 的切换入口需要统一，不建议在多个 UI 位置各自定义自己的切换语义。
