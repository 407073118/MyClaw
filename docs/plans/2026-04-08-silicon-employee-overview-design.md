# Silicon Employee Overview Design

**Date:** 2026-04-08  
**Scope:** `desktop/employees` 的产品概念、主控台关系、私域边界、状态展示与演进方向  
**Status:** Design baseline

## Background

`硅基员工` 不是 OpenClaw-style 的临时 subagent，也不是一次性任务执行器。它是一套长期存在的数字员工系统，每个 `SiliconPerson` 都是独立个体，有自己的头像、title、description、soul、模型、审批策略、skills、workflows、tasklist、schedules、sessions，以及自己的 `currentSession`。

现有 `desktop/employees` 只是伪代码级入口，后续可以直接重写，不需要兼容旧数据。这一点很重要：我们不是在旧实现上做补丁，而是在明确产品范式之后重新定义员工系统。

主聊天也不是“某个员工的私聊页”。它是用户的个人指挥台，是控制平面，不是单个员工的私域会话。用户在主聊天里下达意图、@ 指派员工、查看状态、发起追问和审批，但真正的执行和结果沉淀发生在每个员工自己的私域空间里。

## Goals

1. 把 `SiliconPerson` 定义为长期存在的第一类实体，而不是临时派生对象。
2. 明确主聊天是个人指挥台，员工聊天空间是私域，不混淆两者职责。
3. 统一 `@` 通信原则：主聊天里 `@` 某个员工，消息直接投递到该员工的 `currentSession`。
4. 建立右侧头像栏作为员工运行态投影，靠状态灯展示 idle / running / needs_approval / done / error。
5. 明确“执行结果不自动回传主控制台”，主控台只看状态与提示，不承载完整结果正文。
6. 为后续任务、定时、工作流、技能体系的接入保留清晰的产品边界。

## Non-Goals

1. 不做 OpenClaw-style 的临时 subagent 编排。
2. 不做复杂关联会话搜索，也不在主聊天里自动拼装跨 session 结果。
3. 不把主控制台改造成每个员工的私聊列表。
4. 不要求旧 `desktop/employees` 数据兼容。
5. 不在 overview 阶段定义复杂的执行算法细节，那些留给 runtime 文档。

## Core Concepts

### 1. `SiliconPerson`

`SiliconPerson` 是硅基员工的代码主实体名。每个实体代表一个长期存在的“数字员工”，它不是消息上下文的临时产物，而是可以持续演化的工作体。

一个员工至少包含这些面向：

- 视觉与身份：`avatar`、`title`、`description`
- 角色内核：`soul`
- 能力与执行：`model`、`skills`、`workflows`
- 治理方式：`approvalPolicy`
- 产出与计划：`tasklist`、`schedules`
- 会话空间：`sessions`、`currentSession`

### 2. 主控台与私域

主聊天是控制平面，员工私域是执行平面。两者共享同一个产品，但不共享同一种交互职责。

- 主控台负责“指挥、分发、审批、追问、观察状态”
- 私域负责“执行、沉淀、回看、复用历史结果”

这种分层的核心收益是：用户始终知道自己是在指挥系统，还是在看某个员工自己的工作空间。

### 3. 员工是长期个体

每个硅基员工都有自己的持续身份，不会因为某次任务结束就消失。任务只是其生命周期中的一次运行，session 只是其私域空间中的一个记录单元。这个长期性决定了头像栏、状态灯、会话空间、审批策略都必须围绕“持续存在的员工”来设计。

## Communication Principles

### 1. `@` 只表示投递，不表示建群

主聊天里的 `@某个员工` 的含义很明确：把当前消息投递到那个员工的 `currentSession`。它不是创建一个临时对话群，也不是把员工拉进主聊天里和用户直接多方混聊。

### 2. 没有 `currentSession` 就自动创建默认 session

如果目标员工还没有 `currentSession`，系统自动为其创建默认 session，再把消息写入该 session。这样用户始终可以在主聊天里直接指派，不需要先理解 session 结构。

### 3. 不做复杂关联搜索

通信模型不依赖跨 session 搜索或“找到最相关历史会话”这类复杂逻辑。系统始终把当前指令发到当前会话，避免语义漂移和路由歧义。

### 4. 结果不回灌主控台

员工执行后的完整结果默认留在员工私域，不自动回传到主控制台。主控台只接收状态变化、未读提示、待审批提示，以及可供快速理解的摘要级信号。

### 5. 需要结果时再显式追问

用户如果想继续看结果，可以再次 `@` 该员工追问，也可以点击头像进入员工聊天空间查看完整结果。这保证主控台保持轻量，不被长结果流淹没。

## Status Bar

右侧头像栏是硅基员工运行态的总览，不是会话正文列表。它展示的是每个员工当前的可见状态，而不是完整对话内容。

状态灯语义如下：

- `idle`：空闲
- `running`：运行中
- `needs_approval`：待审批
- `done`：完成
- `error`：错误

同时，头像栏还要显示两个更高层提示：

- 未读提示
- 待审批提示

颜色语义固定为：

- 黄色表示待审批
- 绿色对号表示完成

这样用户无需进入私域就能快速判断哪些员工在跑、哪些人卡在审批、哪些任务已经结束。

## Overall Evolution

硅基员工系统的演进不是从“聊天页”开始，而是从“长期员工实体”开始。建议的总体方向是：

1. 先把 `SiliconPerson` 及其私域会话模型定型。
2. 再把主聊天、头像栏、状态灯、未读/待审批提示接通。
3. 然后把审批、模型绑定、skills、workflows、tasklist、schedules 逐步纳入同一运行时。
4. 最后再补齐更细的员工运营能力，例如配置快照、运行记录、工作流编排和定时触发。

这个顺序的核心原则是：先把“这是一个长期员工系统”讲清楚，再去堆能力，而不是把一堆 agentic 功能包装成临时 subagent。
