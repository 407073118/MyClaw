# 硅基员工 UI 设计

**Date:** 2026-04-08  
**Scope:** `desktop/` 侧的硅基员工交互、工作空间与创建流  
**Status:** Draft

## 背景

当前仓库里的 `employees` 模块只是伪代码级别的占位实现，已经不适合继续在旧结构上补丁式演进。本次设计直接按新产品形态重写为“硅基员工”，其代码主实体统一命名为 `SiliconPerson`，并以此作为 renderer、main process、IPC、持久化与 UI 状态的共同语义中心。

这个产品不是“普通员工列表页”，而是一个以主聊天驱动协作、以员工工作空间承载任务执行与资料管理的双层界面。主聊天叫 `Commander Console`，右侧栏叫 `Silicon Rail`。用户先在主聊天发起协作，再通过右侧栏进入某个硅基员工的工作空间查看详情、追踪任务和管理能力。

## 设计目标

1. 让用户一眼分清“主协作入口”和“员工专属工作空间”。
2. 让右侧栏成为稳定的员工状态入口，而不是另一个聊天列表。
3. 让状态反馈统一、可读、可预测，避免多个地方各自解释进度。
4. 让创建硅基员工的流程清晰分两步：先创建身份，再绑定能力。
5. 让主聊天保持轻量协作痕迹，不把完整执行结果强行回灌到主流对话里。

## 产品模型

- 中文产品名：硅基员工
- 代码主实体：`SiliconPerson`
- 主入口：`Commander Console`
- 侧边入口：`Silicon Rail`
- 工作空间：硅基员工详情页
- 会话模型：每个硅基员工可拥有多个 session

`SiliconPerson` 应当被视为“人”的抽象，而不是一条工具配置。它同时携带身份、能力、审批偏好、默认模型配置和多 session 的协作上下文。UI 上要避免把它展示成简单的模型 preset 或纯任务执行器。

## 整体布局

### 1. Commander Console

主聊天区域保留在页面的核心视觉中心，承担以下职责：

- 接收用户的自然语言指令
- 接收 `@` 投递给当前硅基员工的消息
- 显示轻量级的协作痕迹
- 作为全局启动协作的指挥台

主聊天不承担“完整结果仓库”的职责。对硅基员工的详细执行结果、资料、任务列表、能力信息，都应该放到工作空间里查看。主聊天里只保留足够让用户知道“刚刚发生了什么”的轻量痕迹，例如状态卡片、简短摘要、关键引用、审批结果或跳转提示。

### 2. Silicon Rail

右侧栏固定为竖排头像列表，承担“发现、切换、监控”三件事：

- 每个头像代表一个 `SiliconPerson`
- 头像旁或角标显示当前状态灯
- 支持未读计数
- 支持待审批提示
- 点击头像进入该硅基员工工作空间

Silicon Rail 不是普通联系人列表。它更像一个“值班面板”，让用户快速知道谁在运行、谁在等待审批、谁已经完成、谁发生错误。

### 3. 工作空间

点击某个头像后进入该硅基员工的工作空间。建议的主结构是四个 tab：

- 聊天
- 资料
- 任务
- 能力

这四个 tab 的职责边界要稳定，不要把所有信息混在聊天 tab 里：

- 聊天：该员工自己的 session 列表、历史消息、局部上下文
- 资料：身份信息、描述、模板来源、绑定配置、审批模式
- 任务：待办、执行中、已完成、失败与归档
- 能力：可用能力、模型配置、工具权限、调用限制、默认策略

## 状态体系

### 1. 统一状态灯

右侧栏和工作空间里要共享同一套统一状态语义，不要分别造词。推荐状态包括：

- `idle`
- `running`
- `needs_approval`
- `done`
- `error`

其中：

- 绿色对号表示 `done`
- 黄色表示 `needs_approval`
- `idle` 用于空闲或未激活
- `running` 用于当前执行中
- `error` 用于失败、异常或可恢复问题

如果后续还需要更细状态，例如 `canceling`、`queued`、`syncing`，也应作为细分语义挂在统一状态体系下，而不是把不同区域拆成各自的状态词典。

### 2. 状态展示原则

- 右侧栏优先显示“当前最重要状态”
- 工作空间内可显示更细的阶段说明
- 主聊天只展示足够维持上下文的轻量状态
- 当状态冲突时，以主线程统一状态为准，不让 renderer 本地猜测

## 交互设计

### 1. `@` 投递行为

用户在主聊天输入 `@`，并选择某个硅基员工后，消息应该投递到该员工的 `currentSession`：

- 如果已经存在 `currentSession`，直接投递
- 如果不存在，先创建一个默认 session，再把消息写入其中

这意味着主聊天永远是入口，不是 session 本身的唯一容器。用户在主聊天发出的指令，最终会落到某个硅基员工当前会话上，但 UI 不需要把这一点做得很重。

### 2. session 切换

每个硅基员工允许多个 session。UI 建议支持：

- 查看 session 列表
- 新建 session
- 切换 currentSession
- 对当前 session 发送 `@` 消息

切换 session 时，工作空间聊天 tab 的内容跟随 session 变化，但主聊天的协作痕迹保持简洁，不需要把历史全部重放一遍。

### 3. 主聊天与工作空间的关系

主聊天负责“指挥”和“发起”，工作空间负责“看全量”和“管理细节”。两者的关系应该是：

- 主聊天知道员工状态，但不展示完整执行内容
- 工作空间知道完整 session、任务和能力上下文
- 用户如果想看结果，可以再次 `@` 或点击头像进入工作空间查看

这种分层能避免主聊天被长结果淹没，也能避免 UI 里重复渲染同一段信息。

## 创建流

创建硅基员工建议做成两段式流程，而不是一次性长表单。

### Step 1: 创建身份

第一步只创建身份骨架，用户主要确认：

- 名称
- 头像或标识
- 简短角色描述
- 模板来源
- 默认模型配置来源

### Step 2: 绑定能力

第二步再绑定能力和协作策略，用户主要确认：

- 能力集合
- 是否继承当前模型配置
- 审批模式
- 默认 session 策略
- 是否允许从模板复制现有硅基员工

### 创建体验原则

- 尽量支持从模板复制
- 尽量复用当前模型配置作为初始值
- 创建时显式设置审批模式
- 不要让用户在一屏里同时处理身份、能力和运行参数

如果模板复制与当前模型配置同时存在，UI 应明确告诉用户哪些字段来自模板、哪些来自当前环境、哪些是创建时可覆盖项。

## 详情页设计

### 1. 聊天 tab

聊天 tab 应该展示该硅基员工自己的 session 维度协作信息，包括：

- session 列表
- 当前 session
- 历史消息
- 轻量状态提示
- 必要时的审批入口

这里不应重复主聊天的完整内容；更适合展示“员工自己的工作脉络”。

### 2. 资料 tab

资料 tab 用于稳定展示与编辑硅基员工身份相关的信息：

- 名称、头像、简介
- 模板来源
- 创建时间
- 默认配置摘要
- 审批模式

### 3. 任务 tab

任务 tab 用于聚合具体执行事项：

- 待处理任务
- 运行中任务
- 已完成任务
- 需要审批的任务
- 错误任务

### 4. 能力 tab

能力 tab 用于展示“这个硅基员工会做什么、能访问什么”：

- 可用工具
- 模型能力
- 调用限制
- 默认策略
- 继承自模板还是当前环境

## 空状态、加载态与错误态

- 空状态要明确告诉用户“先创建硅基员工，再开始协作”
- 加载态要尽量短、明确、可预测
- 错误态要区分“可重试”和“需要人工处理”
- 审批等待应在 UI 上突出显示，不能藏在普通错误态里

主聊天中的错误提示要偏协作语义，例如“当前员工需要审批”“执行已暂停”“结果可在工作空间查看”，不要把所有问题都说成系统失败。

## 旧模块处理

旧 `employees` 模块已经被确认可以直接删除或重写，无需迁移兼容。因此 UI 设计上不需要再为旧页面形态、旧字段名、旧交互补兼容层。新的页面应该围绕 `SiliconPerson`、Commander Console 和 Silicon Rail 重新组织。

## 设计结论

这套 UI 的核心不是“员工列表”，而是“主聊天 + 右侧状态 rail + 员工工作空间”的三层协作结构。主聊天负责发起和留下痕迹，右侧栏负责发现和切换，工作空间负责承载完整上下文。只要这三层的职责边界稳定，后续不管 session、审批、能力还是模型配置怎么扩展，UI 都能保持可理解性。
