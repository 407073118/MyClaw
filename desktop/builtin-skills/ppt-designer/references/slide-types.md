# PPT 版式类型详细参考

本文档详细描述所有可用的 slide 版式类型，包括完整的字段定义、JSON 示例和使用指南。

---

## cover — 封面页

### 适用场景
每个演示文稿的第一页。展示标题、副标题、作者和日期信息。视觉上以大标题居中为主，给观众留下第一印象。

### data 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 演示标题，建议 ≤ 15 个中文字，要有冲击力 |
| subtitle | string | 否 | 副标题，通常为部门名称、项目名或补充说明 |
| author | string | 否 | 作者姓名 |
| date | string | 否 | 日期，格式自由，如 "2026年4月" 或 "2026-04-10" |

### JSON 示例
```json
{
  "type": "cover",
  "data": {
    "title": "从数据到决策：Q1 增长复盘",
    "subtitle": "产品与增长部门",
    "author": "张三",
    "date": "2026年4月"
  }
}
```

### 设计建议
- 标题要有吸引力，避免 "XX工作汇报" 这类泛泛写法
- 好的标题示例："从数据到决策"、"三个月，三倍增长"、"下一个增长引擎"
- 差的标题示例："2026年Q1工作汇报"、"张三的述职报告"

## section — 章节过渡页

### 适用场景
用于在演示中划分章节，帮助观众理解内容结构。大字居中显示章节标题和编号，起"呼吸"作用。

### data 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 章节标题，简洁有力 |
| sectionNumber | number | 否 | 章节编号，从 1 开始。如提供，会显示为 "01"、"02" 等样式 |

### JSON 示例
```json
{
  "type": "section",
  "data": {
    "title": "核心业务数据",
    "sectionNumber": 1
  }
}
```

### 设计建议
- 超过 5 页的 deck 必须有至少一个 section 过渡页
- 章节标题尽量控制在 8 个字以内
- 每 3-4 页内容页之后插入一个 section，保持节奏感
- 章节标题应概括后续内容的核心主题，而不是简单编号

## key_points — 要点列表页

### 适用场景
展示 3-6 个核心要点，每个要点配一个图标。适用于策略总结、计划列举、功能介绍等场景。这是最常用的版式。

### data 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 页面标题 |
| points | array | 是 | 要点数组，3-6 个 |
| points[].icon | string | 否 | emoji 图标，如 "🚀"、"📊"。不提供则使用默认圆点 |
| points[].text | string | 是 | 要点内容，控制在一行以内（≤ 20 字） |

### JSON 示例
```json
{
  "type": "key_points",
  "data": {
    "title": "Q2 重点计划",
    "points": [
      { "icon": "🎯", "text": "上线智能推荐系统 v2" },
      { "icon": "🌏", "text": "启动东南亚市场试点" },
      { "icon": "📊", "text": "建设实时数据看板" },
      { "icon": "🤝", "text": "完成 3 家战略合作签约" }
    ]
  }
}
```

### 设计建议
- 严格遵守 6-6 法则：最多 6 个要点，每个要点不超过 15 字
- 要点之间应该平行对等，不要一个长一个短
- 使用有意义的 emoji 图标，与要点内容相关
- 不要在 points 里写完整段落

## metrics — 数据大字报页

### 适用场景
突出 2-4 个核心 KPI 数据，以大数字方式展示。适用于业绩汇报、数据看板、核心指标展示。视觉冲击力强。

### data 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 页面标题 |
| items | array | 是 | KPI 数组，2-4 个 |
| items[].label | string | 是 | 指标名称，如 "营收"、"用户数" |
| items[].value | string | 是 | 指标值，简短格式如 "2.3亿"、"1,240"、"67%" |
| items[].change | string | 否 | 变化值，如 "+23%"、"-15%"、"+25" |
| items[].trend | string | 否 | 趋势方向，只能是 "up" 或 "down" |

### JSON 示例
```json
{
  "type": "metrics",
  "data": {
    "title": "关键业务指标",
    "items": [
      { "label": "营收", "value": "2.3亿", "change": "+23%", "trend": "up" },
      { "label": "新客户", "value": "1,240", "change": "+18%", "trend": "up" },
      { "label": "获客成本", "value": "¥340", "change": "-15%", "trend": "down" },
      { "label": "NPS", "value": "67", "change": "+25", "trend": "up" }
    ]
  }
}
```

### 设计建议
- value 必须是简短的数字表达，绝不能写完整句子
- 好的 value："2.3亿"、"99.9%"、"¥340"、"1,240"
- 差的 value："营收达到了两亿三千万元"
- 2-4 个为最佳，超过 4 个数字视觉会拥挤
- 能用 metrics 版式的尽量不用 key_points，数据更有冲击力
- trend 为 "down" 不一定是坏事（如获客成本下降是好事），颜色由主题决定

## comparison — 左右对比页

### 适用场景
两个方案、时间段、产品的直观对比。以左右两栏展示，对比清晰。适用于方案评估、前后对比、竞品分析。

### data 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 页面标题 |
| leftLabel | string | 是 | 左侧标签，如 "方案A"、"优化前"、"2025 Q4" |
| rightLabel | string | 是 | 右侧标签，如 "方案B"、"优化后"、"2026 Q1" |
| leftPoints | string[] | 是 | 左侧要点列表 |
| rightPoints | string[] | 是 | 右侧要点列表 |

### JSON 示例
```json
{
  "type": "comparison",
  "data": {
    "title": "重构前后对比",
    "leftLabel": "重构前",
    "rightLabel": "重构后",
    "leftPoints": [
      "构建时间 120s",
      "冷启动 8s",
      "单元测试覆盖率 45%",
      "月均 Bug 数 23"
    ],
    "rightPoints": [
      "构建时间 35s",
      "冷启动 2.1s",
      "单元测试覆盖率 89%",
      "月均 Bug 数 5"
    ]
  }
}
```

### 设计建议
- 左右要点数量应该一致或接近
- 每个要点保持简洁，一行以内
- 对比项应该一一对应（左右第 N 条讲同一维度）
- 适合 3-5 条对比项

## closing — 结束页

### 适用场景
演示最后一页。展示感谢语和联系方式。好的结束页不只是"谢谢"，应该有行动号召或关键结论。

### data 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| message | string | 是 | 结束语/感谢语 |
| contact | string | 否 | 联系方式，如邮箱、电话等 |

### JSON 示例
```json
{
  "type": "closing",
  "data": {
    "message": "让我们一起打造下一个增长引擎",
    "contact": "growth-team@company.com"
  }
}
```

### 设计建议
- 结束语不要只写"谢谢"，加入行动号召更有力
- 好的结束语："让数据驱动每一个决策"、"期待各位的反馈与建议"
- 差的结束语："谢谢大家"、"Q&A"
- 联系方式可选，如果是对外演示建议加上
